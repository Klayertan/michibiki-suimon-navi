# QZ1 Satellite Assurance Map

Satellite Assuranceは絶対精度の証明ではありません。同期した2受信機の相対的一致、fix品質、測位継続性、HDOP、衛星数、短期ジャンプ、QZSS・補正関連のNMEA証拠を、農作業向けの保守的な判断材料に変換します。真値測量点がない場合、QZ1とM10の差からどちらが正しいかは判断しません。

## 実装構成

- `js/gnss/nmea-parser.js`: GGAを観測epochとして、RMC/ZDA/GSA/GQGSVを統合。no-fix epoch、チェックサム、生文参照を保持します。
- `js/gnss/gnss-store.js`: 受信機、記録セッション、観測を分離して保持します。
- `js/fields/field-registry.js`: 既存境界に加え、選択したQZ1開始点から終了点までを1枚の圃場として保存します。
- `js/assurance/pairing.js`: 時刻順ソート後、許容時間内の最近傍を一対一・単調に対応付けます。
- `js/assurance/assurance-engine.js`: 説明可能なルール、証拠等級、圃場内に切り抜いた管理セルを計算します。
- `js/assurance/satellite-assurance-controller.js`: UI、Leafletレイヤー、入出力、旧JSON互換を管理します。

## 証拠の意味

| 証拠 | 現在の扱い |
|---|---|
| 緯度・経度・高度 | GGAから取得。真値ではなく受信機の解です。 |
| UTC日時 | RMC/ZDAから取得。日付指定がある場合は指定日を優先し、受信機コールドスタート時の誤日付を避けます。 |
| fix quality | GGAから取得。`2`はDGNSS状態ですが、補正方式をSLASと断定しません。 |
| 衛星数・HDOP | GGA、補助的にGSAから取得します。 |
| QZSS可視 | `$GQGSV`から衛星リストと可視数を取得します。 |
| QZSS測位使用 | GSAの193–202番PRNがある場合に使用証拠として記録します。 |
| SLAS/L1S明示状態 | 現行サンプルには確認できる専用状態文がありません。未確認です。 |
| 生NMEA | 観測ごとに行番号、文種、文を参照として保存します。 |
| DCR/DCX災害メッセージ | 現行サンプルにペイロードまたはデコード文はありません。対応可否はハードウェア試験待ちです。 |

## 対応付け

各受信機の有効座標をUTCミリ秒（なければ時刻のみ）でソートし、同一時刻の重複はメタデータが多い観測を残します。QZ1を先頭から走査し、設定した100–2000 msの許容範囲にある未使用の基準観測のうち、時刻差が最小のものを対応させます。手動時計補正を受信機ごとに適用できます。

この方法は `O(Q log Q + R log R)` のソートと `O(Q + R)` の走査で、異なるサンプリング周波数、欠測、順不同を扱えます。日付のない別セッション同士は誤対応の可能性があるためUIで警告します。

## スコアと証拠等級

ペアのAssurance scoreは5点刻みで表示し、利用できる証拠だけで再正規化します。

| 寄与 | 重み | 内容 |
|---|---:|---|
| 補正・補強証拠 | 20 | 明示状態、またはDGNSS状態。推定と確認済みを区別します。 |
| fix | 10 | 有効fix。 |
| HDOP | 15 | 1以下を最良、10以上を最低とする区分線形評価。 |
| 衛星数 | 5 | 9以上を最良とする段階評価。 |
| 受信機間一致 | 20 | 運用プロファイル別しきい値。正しさではなく相対的一致です。 |
| 継続性 | 15 | 全epochに対する有効fix率。no-fixを消しません。 |
| 短期安定性 | 15 | 時間差と運用速度から許されない位置ジャンプを検出します。 |

セルの証拠等級は、ペア数、観測時間、時刻の完全性、メタデータ充足率、補正証拠からA–Dで表します。プロファイルの最低等級、最低ペア数、スコア、明示補強の要否を満たす場合だけ緑になります。欠測セルは灰色のままです。SIMULATEDデータを含む結果は紫のSIM分類となり、緑・黄・赤の運用判断にはなりません。

## 空間表現と性能

5 mの圃場内クリップグリッドを既定とし、個別ペア線と生点を任意表示します。補間やヒートマップは行わないため、測定していない場所に証拠があるようには見せません。セル探索は行・列インデックスで平均`O(1)`、集計はペア数に対して`O(P)`です。

数万観測まではメインスレッドで扱える設計です。解析時だけ再計算し、表示レイヤーの切り替えでは再計算しません。10万点級や頻繁なライブ再計算が必要になった時点でWeb Workerと描画間引きを検討します。

## JSON v2

`schemaVersion: "2.0.0"`の書き出しには以下を含みます。

- `gnss.receivers`: 機種・役割・能力と根拠
- `gnss.sessions`: ソース、捕捉日、期待周期、シミュレーション表示、パーサ要約
- `gnss.observations`: 生観測とメタデータ
- `fields`: 圃場境界、点範囲の開始・終了・順序・元セッション
- `assurance.configuration`: セッション選択、時計補正、許容時間、プロファイル、セルサイズ
- `assurance.lastRun`: コンパクトな対応ペア、セル結果、サマリー、計算版
- `calculationVersions`: 再計算互換性を判断する版

派生結果は監査用に保存しますが、読み込み時は現在の計算版で再計算します。旧測量ポイント配列と`paddy-intelligence.v1`は読み込み可能です。旧データは受信機ID、生NMEA、完全な時刻を復元できないため、Assurance用途では「受信機不明」として扱います。

## ハードウェア確認

実運用デモ前に、同じアンテナ条件・同時刻でQZ1とM10を記録し、以下を確認します。

1. 両受信機のUTC日付、時刻、出力周期、時刻オフセット。
2. QZ1のGGA/GSA/GSV/RMC/ZDAと、追加の独自文をすべて保存。
3. fix quality `2`の補正方式を示す、仕様書または明示的な受信機出力。
4. QZSS PRNがGSAで使用衛星として現れる条件。
5. 静止10分、外周複数周、建物・樹木・水路際での再現性。
6. M10側の同等メタデータとアンテナ位置差。
7. 既知の測量基準点が使える場合だけ、絶対誤差を別指標として追加。
8. DCR/DCX、災害情報、生L1Sペイロードをシリアル出力できるか、設定・ファームウェア・仕様書で確認。

現時点の災害モード判定は「ハードウェア試験なしでは不明」です。初期MVPにデコーダや受信可能という表示は含めません。将来は`DisasterAlertAdapter`境界、明確なSIMULATED fixture、ローカル圃場アクションUIを先に用意し、実機出力が確認できた場合だけデコーダを接続します。
